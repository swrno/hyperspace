# Application Management

An **Application** in `hypr` represents a configured chatbot designed for integration into a specific product. Each application specifies a unique system prompt, a selected LLM chain, sampling parameters (temperature, max tokens), and a set of linked Knowledge Bases.

---

## Authentication Schemes

Integrating external client apps with `hypr` uses a two-level security scheme that separates the owner's credentials from the public-facing SDK.

```
                  [ Owner dashboard ]
                           │
                  Firebase Authentication
              (Bearer Token -> verifyToken)
                           │
             ┌─────────────┴─────────────┐
             ▼                           ▼
  [ Manage Applications ]      [ Manage API Keys ]
   Generate appId, clientId      Generate account-wide keys
   (e.g., app_6c7d8e...)         (e.g., sk_live_4a5b6c...)
             │                           │
             └─────────────┬─────────────┘
                           ▼
                 [ Public hypr-sdk ]
              apiKey + appId + clientId
```

### Identifier Breakdown

1. **`appId`** (`app_<hex>`):
   - A unique identifier generated per application instance. Used to scope queries and sync end-user conversation logs.
2. **`clientId`**:
   - The app owner's Firebase user ID (`uid`). This links the application to the owner's billing and account dashboard.
3. **`apiKey`** (`sk_live_<hex>`):
   - An account-wide API key generated by the owner under their profile. 
   - **Important**: API keys are not scoped to a single application. Any active API key owned by a user can authenticate SDK queries for *any* application that user owns.

## Data Structure (MongoDB)

End-user interactions are tracked across two MongoDB collections. This separates high-frequency message streams from static user profile statistics.

### 1. `app_users`
Tracks individual user activity metrics scoped to a specific application.
```ts
interface AppUser {
  _id: ObjectId;
  appId: string;        // Scoped app identifier (app_...)
  userId: string;       // Opaque end-user identifier from the client app
  createdAt: string;    // ISO Date of first query
  lastActiveAt: string; // ISO Date of most recent query
  turnCount: number;    // Cumulative message turns
}
```

### 2. `conversations`
Stores conversation history, partitioned by user sessions.
```ts
interface Message {
  id: number;          // Timestamp-based message ID
  role: 'user' | 'assistant';
  content: string;     // Polished answer text
  reasoning?: string;  // Extracted reasoning block (if applicable)
  timestamp: string;   // ISO Date
  sessionId: string;   // Identifier for the chat thread
}

interface Conversation {
  _id: ObjectId;
  appId: string;
  userId: string;
  sessionId: string;
  messages: Message[];
  createdAt: string;
  updatedAt: string;
}
```

## End-User Chat History Retrieval

Workspace owners can view conversation logs in the application playground's **"End-User Chat History"** tab. This interface queries the `/api/app-users` endpoint to pull:
- A list of active end-users ordered by `lastActiveAt` (most active first).
- Active conversation threads and messages scoped to the selected user.

::: warning Security Constraint
The dashboard fetches this history using the owner's **Firebase Authentication token** (`Authorization: Bearer <idToken>`). SDK-level API keys cannot access user history queries, protecting end-user privacy from client-side credentials.
:::
